## Context

Paperland renders markdown content (Q&A answers, paper abstracts) in the browser using `marked` v17 with `marked-katex-extension` for math formulas. A custom `normalizeDelimiters()` function pre-processes the markdown to convert `\(...\)` and `\[...\]` delimiters to `$`/`$$` before passing to `marked`, using regex with code-block protection.

The current setup has two structural problems:
1. `marked` frequently fails on valid markdown — particularly around nested structures, complex tables, and edge cases where math delimiters interact with other syntax
2. The `normalizeDelimiters()` regex approach is fragile — it protects code blocks but misses other contexts (e.g., delimiters inside HTML attributes, nested fenced blocks)

The text highlighting system (`useHighlight.ts`) operates entirely on the rendered DOM — it walks text nodes and KaTeX elements, computes global offsets, and wraps ranges in `<mark>` tags. It is parser-agnostic as long as:
- KaTeX elements have `.katex` / `.katex-display` classes (always true, generated by KaTeX itself)
- The rendered HTML is valid DOM that can be traversed

## Goals / Non-Goals

**Goals:**
- Replace `marked` with `markdown-it` for more reliable, spec-compliant markdown parsing
- Support all four math delimiter styles natively (`$`, `$$`, `\(`, `\)`, `\[`, `\]`) via plugin
- Maintain full GFM support (tables, strikethrough, task lists)
- Keep the highlight system fully functional without modifications
- Remove the fragile `normalizeDelimiters()` workaround

**Non-Goals:**
- Syntax highlighting for code blocks (not currently supported, not adding it now)
- Server-side markdown rendering
- Changing the highlight system's offset model or storage format
- Adding new markdown extensions beyond what GFM + math provide

## Decisions

### Decision 1: Use `markdown-it` as the replacement parser

**Choice**: `markdown-it` over `remark`/`unified` or `micromark`

**Rationale**:
- `markdown-it` is CommonMark-compliant with a battle-tested, mature plugin ecosystem
- GFM features (tables, strikethrough) are built-in — just enable `html: true, linkify: true, typographer: false, breaks: true`
- Plugin architecture is simpler than `unified`'s AST transform pipeline — we just need HTML output, not an AST
- `@traptitech/markdown-it-katex` provides native support for all four math delimiter styles without manual preprocessing
- Drop-in replacement: `md.render(content)` produces HTML string, same as `marked.parse(content)`
- `remark`/`unified` would be over-engineering — we don't need AST manipulation, just reliable HTML rendering

### Decision 2: Use `@traptitech/markdown-it-katex` for math

**Choice**: `@traptitech/markdown-it-katex` over `markdown-it-texmath` or `markdown-it-mathjax`

**Rationale**:
- Directly integrates KaTeX (which we already depend on) — no new rendering engine
- Handles `$`, `$$`, `\(`, `\)`, `\[`, `\]` natively at the parser level, eliminating `normalizeDelimiters()` entirely
- Processes math delimiters during tokenization, so they're correctly distinguished from code blocks and other syntax
- Same KaTeX CSS import still works (`katex/dist/katex.min.css`)
- Supports `throwOnError: false` for graceful degradation of invalid LaTeX

### Decision 3: No changes to the highlight system

**Choice**: Keep `useHighlight.ts` completely untouched

**Rationale**:
- The highlight system's `buildTextSegments()` identifies KaTeX elements by CSS class (`.katex`, `.katex-display`), which KaTeX always generates regardless of which markdown parser feeds it
- Text node traversal works on any valid DOM tree
- Content hash is computed from raw markdown source (before parsing), so parser choice doesn't affect it
- Offset-based text matching uses `extractText()` which compares rendered text content — same text produces same offsets regardless of HTML structure differences between parsers

### Decision 4: Instantiate `markdown-it` as a module-level singleton

**Choice**: Create and configure the `markdown-it` instance once, outside the component

**Rationale**:
- Same pattern as current `marked.use()` / `marked.setOptions()` at module level (lines 36-37 of MarkdownContent.vue)
- Avoids recreating the parser on every render
- Configuration: `{ html: false, breaks: true, linkify: true }` + `.use(mk, { throwOnError: false })`

## Risks / Trade-offs

**[Risk] Rendered HTML differs slightly between parsers** → The highlight system matches stored text against rendered `textContent`, not HTML structure. Since `textContent` strips tags, minor HTML differences (e.g., `marked` outputs `<br>` vs `markdown-it` outputs `<br />`) won't affect highlights. However, if `markdown-it` produces semantically different text flow (e.g., different whitespace in rendered text), some existing highlights may fail offset verification and silently disappear. Mitigation: the highlight system already handles this gracefully (skips non-matching highlights).

**[Risk] Plugin compatibility with Bun bundler** → `markdown-it` and `@traptitech/markdown-it-katex` are standard npm packages. Vite handles bundling, and both packages use standard CJS/ESM exports. Low risk — these are widely used packages.

**[Risk] Bundle size change** → `markdown-it` (~100KB min) is comparable to `marked` (~70KB min). The net increase is minimal since we're removing `marked` + `marked-katex-extension` and adding `markdown-it` + `@traptitech/markdown-it-katex`. KaTeX bundle stays the same.
